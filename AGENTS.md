# AGENTS.md

## 1. Overview

- This repository contains the AuraBeam project: a Python-based nighttime vehicle localization, sensor-fusion, evaluation, and Hardware-in-the-Loop workflow for adaptive driving beam experiments.
- This file defines strict rules for AI coding agents and human contributors working on this codebase.
- Follow these instructions before making code, configuration, report, firmware, or experiment changes.

## 2. Repository Architecture

- `src/aura_beam/`: core reusable application logic.
  - `detector_ensemble.py`: detector loading, YOLO inference, and ensemble fusion.
  - `pseudo_radar.py`: depth surrogate logic.
  - `sensor_fusion.py`: Kalman filters, tracking, and observation policies.
  - `zone_logic.py`: LED-grid mapping and control decisions.
  - `serial_manager.py`: Arduino serial communication.
- `scripts/runtime/`: runtime entry points.
- `scripts/evaluation/`: experiment, metric, aggregation, figure, and sweep scripts.
- `configs/`: JSON experiment suites and occlusion interval definitions.
- `tests/`: pytest tests.
- `firmware/`: Arduino firmware. Keep the canonical firmware in `firmware/arduino_8x8_matrix.ino`.
- `report/`: LaTeX manuscript and bibliography files.
- `artifacts/`, `figs/`, `statistic/`, `metrics_ultralytics/`: generated or curated experiment outputs used for analysis and reporting.
- `model/`, `demo_video/`, and scenario folders such as `fogging_1/`, `rain_3/`, `thunder_1/`: model weights, videos, frames, and annotations required by experiments.
- Root-level Python files such as `main.py`, `evaluate_metrics.py`, and `aggregate_results.py` are compatibility wrappers. Do not add new core logic to these wrappers.

Dependency rules:

- Core logic belongs in `src/aura_beam/`.
- Evaluation-only logic belongs in `scripts/evaluation/`.
- Runtime orchestration belongs in `scripts/runtime/`.
- Core modules must not import from `scripts/`, `tests/`, `report/`, or `artifacts/`.
- Scripts may import from `src/aura_beam/`.
- Tests may import from both `src/aura_beam/` and scripts.
- Do not introduce circular imports.
- Do not make source code depend on generated files unless the dependency is explicit in a config file.

## 3. Code Conventions

- Use Python 3 syntax.
- Use `snake_case` for variables, functions, methods, modules, and files.
- Use `PascalCase` for classes.
- Use `UPPER_SNAKE_CASE` for constants.
- Keep public function names stable unless the change is explicitly a breaking API change.
- Prefer domain-oriented modules over file-type-oriented modules. Example: tracking logic goes in `sensor_fusion.py`, not in a generic `utils.py`.
- Functions should have one responsibility. Split functions that mix loading, inference, metric calculation, file output, and plotting.
- Keep functions short enough to review. As a rule, refactor functions longer than about 80 lines unless the existing style clearly requires otherwise.
- Use type hints for new public functions and non-trivial helper functions.
- Avoid broad `except Exception` blocks. Catch specific exceptions and include actionable error messages.
- Validate file paths, config keys, and required model/video inputs before long experiment runs.
- Do not silently ignore failed detections, missing annotations, serial errors, or malformed JSON configs.
- No formatter or linter configuration is currently present in the repo. Match the existing style, keep imports organized, and run tests after changes.

## 4. Important Rules (Critical)

DO:

- Keep changes scoped to the requested task.
- Preserve existing command-line interfaces and config schemas unless the task explicitly requires a change.
- Add or update tests for new behavior in `src/aura_beam/`, `scripts/runtime/`, or `scripts/evaluation/`.
- Update relevant docs when changing setup, experiment commands, config behavior, or report-generation steps.
- Use `configs/` for experiment parameters instead of hardcoding values in scripts.
- Put generated experiment outputs under `artifacts/` or an existing output folder.
- Keep root-level wrappers thin.
- Check large model, video, and result files before adding them to version control.

DON'T:

- Do not modify unrelated files.
- Do not break existing APIs, config keys, output CSV schemas, or root wrapper behavior without documenting the migration.
- Do not hardcode secrets, absolute machine-specific paths, serial ports, model paths, or dataset paths.
- Do not commit credentials, API keys, tokens, or private data.
- Do not edit generated results by hand when they can be regenerated by a script.
- Do not put manuscript-only logic in `src/aura_beam/`.
- Do not put core algorithm logic in root wrappers.
- Do not introduce circular dependencies.
- Do not delete experiment artifacts, configs, or report assets unless the task explicitly asks for cleanup.

## 5. Build & Run Instructions

Install dependencies on Windows PowerShell:

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
```

If PowerShell blocks activation:

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\venv\Scripts\Activate.ps1
```

Quick environment checks:

```powershell
python --version
python run_experiment_suite.py --help
python aggregate_results.py --help
```

Run Q2 experiment batches:

```powershell
.\venv\Scripts\Activate.ps1
.\run_q2_a1_a4.ps1
.\run_q2_b1_b2.ps1
.\run_q2_c1_c4.ps1
```

Run a partial batch:

```powershell
.\run_q2_b1_b2.ps1 -StartRun 3 -EndRun 5
```

Aggregate results manually:

```powershell
python aggregate_results.py --results-root artifacts/results_q2_a1_a4 --output-csv artifacts/tables/q2_a1_a4_results.csv
python aggregate_results.py --results-root artifacts/results_q2_b1_b2 --output-csv artifacts/tables/q2_b1_b2_results.csv
python aggregate_results.py --results-root artifacts/results_q2_c1_c4 --output-csv artifacts/tables/q2_c1_c4_results.csv
```

There is no separate build system currently configured. For this Python project, "build" means installing dependencies and verifying that scripts/tests run.

## 6. Test Instructions

Run all tests:

```powershell
python -m pytest
```

Run a single test file:

```powershell
python -m pytest tests/test_project_structure.py
```

Run a single test:

```powershell
python -m pytest tests/test_project_structure.py::test_core_package_exists
```

Testing rules:

- Tests must pass before merging.
- Add tests for new core logic, config parsing, metric calculations, and behavior changes.
- For bug fixes, add a regression test when practical.
- No formal coverage threshold is currently configured. Do not reduce meaningful coverage of touched code.
- If a test requires unavailable large files, models, hardware, or videos, mark it clearly and keep a lightweight unit test for the logic path.

## 7. CI/CD

- No CI/CD configuration is currently visible in this repository.
- Before merging, contributors must run the relevant local checks manually.
- Required local checks for code changes:

```powershell
python -m pytest
```

- Required local checks for experiment workflow changes:

```powershell
python run_experiment_suite.py --help
python aggregate_results.py --help
```

- A change should be considered failing if tests fail, required scripts cannot start, JSON configs are invalid, or expected output schemas change without documentation.

## 8. Examples / Patterns

Good: keep experiment parameters in config files.

```python
config_path = Path(args.config)
suite = json.loads(config_path.read_text(encoding="utf-8"))
tau_conf = suite["tracking"]["tau_conf"]
```

Bad: hardcode experiment parameters inside reusable logic.

```python
tau_conf = 0.35
video_path = "C:/Users/ADMIN/Desktop/private_video.mp4"
```

Good: keep root wrappers thin.

```python
from scripts.evaluation.aggregate_results import main

if __name__ == "__main__":
    main()
```

Bad: add new aggregation behavior directly to a root wrapper.

```python
# Do not place new experiment logic here.
```

Good: raise actionable errors for missing inputs.

```python
if not model_path.exists():
    raise FileNotFoundError(f"Missing model weights: {model_path}")
```

Bad: continue after missing required inputs.

```python
if not model_path.exists():
    print("missing model")
```

Good: keep core code independent from generated artifacts.

```python
from aura_beam.sensor_fusion import KalmanFilter3D
```

Bad: import report or artifact files from core modules.

```python
from artifacts.tables import final_evaluation_results
```
